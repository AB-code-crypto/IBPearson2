Сразу поправлю один момент: сейчас у тебя **не 50**, а **30** кандидатов в shortlist после первого Пирсона. Порог по Пирсону действительно **0.80**, а потом в decision layer лучший `similarity_score` должен быть не ниже **0.5**. Это прямо задано в `ts_config.py`: `PEARSON_SHORTLIST_MIN_CORRELATION = 0.80`, `PEARSON_SHORTLIST_TOP_N = 30`, `DECISION_MIN_BEST_SIMILARITY_SCORE = 0.5`. Там же задано, что в forecast берутся только **5** лучших similarity-кандидатов, а decision требует минимум **5** кандидатов после similarity и минимум **5** кандидатов в forecast. ([GitHub][1])

Текущая логика у тебя идёт так.

**1. В начале каждого нового часа runtime определяет, с какими историческими часами вообще можно сравниваться.**
Это делает `PearsonLiveRuntime._start_new_hour()`: он вычисляет `allowed_hour_slots = resolve_allowed_hour_slots(self.current_hour.hour_slot_ct)`, потом грузит из `prepared.sqlite3` все prepared-часы только для этих `hour_slot_ct` и только строго раньше текущего часа через `load_prepared_hours_by_slots(..., before_hour_start_ts_ct=self.current_hour.hour_start_ts_ct)`. То есть сначала берётся **не вся история подряд**, а только исторические часы из разрешённых CT-слотов. ([GitHub][2])

**2. Сейчас `resolve_allowed_hour_slots()` работает не по “same hour only”, а по схеме `SEARCH_SLOT_GROUPS_MARKET_PHASES_CT`.**
Это означает:

* для постмаркета используются слоты `[17,18,19,20,21,22,23,0,1]`,
* для премаркета `[2,3,4,5,6,7]`,
* для часа открытия только `[8]`,
* для основной сессии `[9,10,11,12,13,14,15]`,
* для клиринга `16` список пустой `[]`.
  Именно эта схема сейчас активна, потому что внизу файла выбрано `SEARCH_SLOT_GROUPS = SEARCH_SLOT_GROUPS_MARKET_PHASES_CT`. ([GitHub][3])

**3. Внутри часа робот не считает стратегию сразу, а ждёт нужного окна по времени.**
В `ts_config.py` сейчас:

* `PEARSON_BAR_INTERVAL_SECONDS = 5`,
* `PEARSON_EVAL_START_MINUTE = 30`,
* `PEARSON_EVAL_END_MINUTE = 50`.
  Это превращается в окно расчёта: `current_bar_count >= 360` и `< 600`, то есть примерно с 30-й до 50-й минуты часа. Пока это окно не открылось, live-runtime просто копит текущий префикс часа и не запускает ranking/forecast/decision. ([GitHub][1])

**4. Что именно копится в текущем часу.**
В `PearsonCurrentHour.add_bar()` текущий ряд `x` строится как нормализованное движение относительно первого mid-open часа:
`x_i = mid_close_i / mid_open_0 - 1`, где `mid_open_0 = (ask_open + bid_open) / 2` первого бара часа, а `mid_close = (ask_close + bid_close) / 2` для каждого следующего бара. То есть стратегия сравнивает не абсолютную цену, а форму движения часа в относительных величинах. ([GitHub][4])

**5. Важный фильтр до любой аналитики: текущий час должен быть валидным.**
В `PearsonLiveRuntime._append_bar_to_current_hour()` час помечается невалидным, если внутри него обнаружена дырка по времени (`bar_time_ts > expected_next_bar_time_ts`) или если в текущем баре есть `NULL` по `ask_open`, `bid_open`, `ask_close`, `bid_close`. Если час стал невалидным, `x` всё ещё продолжают копить для отладки, но сам расчёт стратегии дальше не выполняется, потому что основной pipeline запускается только при `self.current_hour_valid and self._is_search_window_active()`. ([GitHub][2])

**6. Первый аналитический слой — грубый shortlist по Пирсону.**
Когда окно поиска активно, `PearsonLiveRuntime.on_closed_bar()` либо один раз вызывает `self.current_hour.initialize_candidates()`, либо потом на каждом новом баре делает `self.current_hour.update_candidates_for_last_bar()`. Это нужно, чтобы не пересчитывать всё с нуля: у каждого исторического кандидата хранится `sum_xy`, и корреляция обновляется инкрементально. После этого вызывается `self.current_hour.get_ranked_candidates(min_correlation=self.min_correlation, top_n=self.top_n)`. Здесь как раз и применяются твои текущие настройки: минимум корреляции **0.80** и максимум **30** лучших кандидатов. ([GitHub][2])

**7. Что такое “кандидат после Пирсона”.**
Это уже не весь prepared-hour, а объект с полями вроде `hour_start_ts`, `hour_start_ct`, `hour_slot_ct`, `contract`, `correlation`. То есть после первого слоя ты получаешь не решение о сделке, а просто shortlist исторических часов, которые по форме текущего префикса сейчас наиболее похожи по обычному Пирсону. ([GitHub][4])

**8. Второй слой — `similarity`, который пересортировывает shortlist уже более осмысленно.**
`PearsonLiveRuntime._rank_similarity_candidates()` берёт не всю историю, а только shortlist после Пирсона и передаёт его в `rank_prepared_candidates_by_similarity(current_values=self.current_hour.x, prepared_hours=shortlist_prepared_hours, min_required_pearson=None)`. То есть второй слой не фильтрует историю с нуля, а работает уже только по тем кандидатам, которых пропустил первый Пирсон. ([GitHub][2])

**9. Какие метрики входят в `similarity score`.**
В `candidate_scoring.py` для каждого префикса считаются:

* `pearson`,
* `range`,
* `net_move`,
* `mean_abs_diff`,
* `path_efficiency`.
  Дальше сравниваются расстояния между текущим и историческим префиксом, часть метрик переводится из “чем больше, тем лучше” в score `0..1`, часть — из “чем меньше расстояние, тем лучше” тоже в `0..1`, а потом считается `final_score` как взвешенное среднее. ([GitHub][5])

**10. Текущие веса и пороги similarity.**
Сейчас в `ts_config.py` стоят:

* `SIMILARITY_PEARSON_SCORE_ZERO_AT = 0.50`,
* `SIMILARITY_PEARSON_SCORE_ONE_AT = 1.00`,
* для расстояний по `range/net_move/mean_abs_diff/path_efficiency` порог обнуления сейчас везде `0.50`,
* веса: `pearson=4.0`, `range=2.0`, `net_move=2.0`, `mean_abs_diff=2.0`, `efficiency=1.0`.
  То есть второй слой по-прежнему сильнее всего доверяет Пирсону, но уже не слепо: он дополнительно штрафует кандидатов, у которых похожая корреляция, но другая амплитуда, другой итоговый сдвиг, другая “рваность” пути и другая эффективность движения. ([GitHub][1])

**11. После similarity берутся только лучшие 5 кандидатов для прогнозного слоя.**
Это делает `PearsonLiveRuntime._build_forecast_summary()`: он режет список так — `selected_similarity_candidates = ranked_similarity_candidates[:FORECAST_TOP_N_AFTER_SIMILARITY]`, а `FORECAST_TOP_N_AFTER_SIMILARITY` сейчас равен `5`. То есть к forecast не идут все 30 кандидатов после Пирсона и даже не все после similarity, а только самая верхушка similarity-ранжирования. ([GitHub][2])

**12. Что делает forecast.**
В `candidate_forecast.py` для каждого выбранного исторического кандидата строится `future_path` от текущей точки до конца часа. Он считается как относительное изменение исторического ряда `y` от `current_bar_index` до конца часа. Потом по группе кандидатов собираются:

* `candidate_count`,
* `positive_count`, `negative_count`, `flat_count`,
* `positive_ratio`, `negative_ratio`,
* `mean_final_move`, `median_final_move`,
* `mean_max_upside`, `median_max_upside`,
* `mean_max_drawdown`, `median_max_drawdown`,
* `mean_future_path`, `median_future_path`.
  Это уже прогнозный слой в человеческом смысле: не “на что текущий час похож”, а “что обычно происходило дальше у лучших похожих часов”. ([GitHub][6])

**13. Дальше идёт formal decision layer.**
`PearsonLiveRuntime._build_decision_result()` просто вызывает `evaluate_decision_layer(ranked_similarity_candidates, forecast_summary)`. А сам `candidate_decision.py` строит `diagnostics`, где есть:

* `similarity_candidate_count`,
* `forecast_candidate_count`,
* `best_similarity_score`,
* `mean_final_move`,
* `median_final_move`,
* `mean_direction`,
* `median_direction`,
* `positive_ratio`,
* `negative_ratio`,
* `mean_max_upside`,
* `mean_max_drawdown`.
  И уже на этом основании возвращает словарь вида `{"decision": "LONG"/"SHORT"/"NO_TRADE", "reason": "...", "diagnostics": {...}}`. ([GitHub][7])

**14. Текущие правила decision layer по шагам.**
Чтобы появился реальный `LONG` или `SHORT`, сейчас должны одновременно выполняться такие условия:

1. `len(ranked_similarity_candidates) >= DECISION_MIN_SIMILARITY_CANDIDATES`, а это сейчас **5**;
2. `forecast_summary` не `None`;
3. `forecast_summary["candidate_count"] >= DECISION_MIN_FORECAST_CANDIDATES`, это тоже **5**;
4. `best_similarity_score >= 0.5`;
5. если включено требование согласованности, то `mean_direction == median_direction`, и это требование сейчас включено;
6. направление не должно быть `FLAT`;
7. `abs(mean_final_move) >= 0.0005`;
8. `abs(median_final_move) >= 0.0005`;
9. для `LONG` нужно `positive_ratio >= 0.60`;
10. для `SHORT` нужно `negative_ratio >= 0.60`.
    Фильтр adverse move в коде есть, но сейчас он отключён: `DECISION_USE_ADVERSE_MOVE_FILTER = False`. ([GitHub][1])

**15. Что именно означает `LONG` или `SHORT` в текущей логике.**
Не “лучший исторический кандидат был вверх”, а именно вот что: группа из лучших similarity-кандидатов дала согласованный прогноз, у неё достаточно сильное ожидаемое движение, достаточная доля кандидатов смотрит в одну сторону, и лучший similarity-кандидат сам по себе тоже не слишком слабый. Только тогда decision layer возвращает торговое решение, а не `NO_TRADE`. ([GitHub][7])

**16. Как решение доходит до реального входа.**
В `main.py` создаются `PearsonLiveRuntime` и `DecisionOrderExecutor`, а дальше они подключаются в realtime-контур через `load_realtime_task(...)`. То есть live-бар приходит, runtime строит `snapshot`, а executor уже работает с этим snapshot. Перед стартом realtime ещё один раз синхронизируется prepared DB через `run_prepared_sync_once(...)`, чтобы убрать гонку между live и prepared-данными. ([GitHub][8])

**17. Условия, при которых executor вообще может открыть сделку.**
В `DecisionOrderExecutor.on_snapshot()` он сначала синхронизирует runtime-state, потом — если `self.enabled` включён — сначала проверяет выход, потом вход. На вход `_can_enter(snapshot)` требует:

* чтобы `self.state.position_side is None`, то есть позиции сейчас нет;
* чтобы `snapshot.decision_calculated` был `True`;
* чтобы `snapshot.decision_result` не был `None`;
* чтобы `snapshot.decision_result["decision"]` был именно `LONG` или `SHORT`;
* чтобы в этом же часу уже не было попытки входа: `self.state.last_entry_attempt_hour_start_ts != snapshot.hour_start_ts`.
  Именно это и даёт твою текущую политику “не более одного входа в час”. ([GitHub][9])

**18. Какой именно ордер отправляется при входе.**
Если `_can_enter()` возвращает `True`, executor:

* создаёт запись сделки через `create_trade(...)`,
* пишет event `SIGNAL_ACCEPTED`,
* берёт активный `local_symbol`,
* строит `order_ref`,
* для `LONG` вызывает `order_service.buy_market(...)`,
* для `SHORT` вызывает `order_service.sell_market(...)`,
* ждёт исполнение с `wait="done"`,
* затем ставит статусы `ENTRY_SUBMITTED` и `OPEN`.
  То есть сейчас вход — это именно market order, а не отложенный лимитный сценарий. Количество контрактов сейчас `trading_order_quantity = 1`, `TIF = DAY`, таймауты — `accept=5s`, `done=60s`, а флаг `trading_enable_order_execution` сейчас включён (`True`). ([GitHub][9])

**19. Когда сделка закрывается.**
Хотя ты спросил “до открытия”, напомню и этот последний шаг для цельной картины: `_should_exit(snapshot)` возвращает `True`, если позиция открыта и либо:

* это тот же час, но `snapshot.current_bar_index >= self.exit_bar_index`,
* либо уже начался новый час.
  В `config.py` сейчас `trading_exit_bar_index = 717`, то есть выход делается за 10 секунд до конца часа для 5-секундных баров. Потом отправляется market-order в обратную сторону. ([GitHub][9])

Если совсем коротко свести текущую боевую цепочку в одну строку, то она сейчас такая:

**новый час → `allowed_hour_slots` по `MARKET_PHASES_CT` → загрузка prepared-часов только из этих CT-слотов → с 30-й по 50-ю минуту считаем shortlist по Пирсону (`corr >= 0.80`, top 30) → на shortlist считаем `final_score` по 5 similarity-фильтрам → берём top 5 в forecast → decision layer требует минимум 5 кандидатов, `best_similarity_score >= 0.5`, согласованность mean/median, достаточную силу движения и directional ratio 0.60 → если решение `LONG/SHORT`, executor один раз в час может открыть market-сделку.** ([GitHub][1])
